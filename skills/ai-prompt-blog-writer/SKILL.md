# 기술 블로그 작성

## 핵심 원칙

> 이 스킬의 목표는 "설명형 기술 글"을 만드는 것이다.
> 최신 비교 리뷰나 마케팅 문구보다, 개념 정리와 규칙 설명이 먼저다.

### 해야 할 것

- 주제를 바로 정의한다
- 왜 중요한지, 언제 쓰는지 먼저 설명한다
- 핵심 규칙이나 기법을 번호 매긴 섹션으로 나눈다
- 각 항목마다 예시 코드, 나쁜 예/좋은 예, 적용 전/후를 붙인다
- 글 마지막에 참고 자료를 남긴다
- 사용자가 준 메모, 경험, 코드가 있으면 그 맥락을 최대한 유지한다
- web-search를 활용해 다른 블로그나 문서들을 인용한다.

### 하지 말아야 할 것

- 개인 경험이 없는데 직접 체험한 것처럼 쓰지 않는다
- 과장된 표현이나 홍보성 문구를 쓰지 않는다
- 사용자의 거친 문체를 과하게 세련되게 바꿔 원래 의도를 지우지 않는다
- 최신 정보가 필요한 주제를 확인 없이 단정하지 않는다

## 사용 방식

이 스킬은 아래 요청에서 특히 잘 맞는다.

- "프롬프트 엔지니어링 개념 정리 글 써줘"
- "안티패턴 중심으로 기술 블로그 글 초안 만들어줘"
- "이 주제를 번호 매긴 정리형 글로 바꿔줘"
- "Jekyll `_posts`용 markdown 글로 작성해줘"

전체 초안을 쓰기 전에는 `references/blog-style.md`를 먼저 읽는다.

## 입력 해석

사용자 요청에서 아래 항목을 추출한다.

| 항목 | 기본값 | 값 |
|---|---|---|
| 글 유형 | guide | `guide` / `anti-pattern` / `pattern` / `concept` |
| 길이 | medium | `short` / `medium` / `long` |
| 독자 수준 | intermediate | `beginner` / `intermediate` / `advanced` |
| 예시 형식 | prompt-and-code | `prompt-only` / `code-only` / `prompt-and-code` |
| 최신성 확인 | auto | `off` / `auto` / `strict` |
| 산출물 범위 | full-draft | `ideas` / `outline` / `full-draft` / `revise` |

옵션이 명시되지 않으면 사용자 요청과 저장소 스타일을 기준으로 합리적으로 추정한다.

### 최신성 규칙

- 모델 비교, 프레임워크 버전, 제품 기능처럼 바뀔 수 있는 주제는 `auto` 이상으로 본다
- 이런 주제는 공식 문서나 1차 자료를 먼저 확인한 뒤 쓴다
- 본문에 날짜를 넣어야 하면 상대 표현 대신 절대 날짜를 쓴다

## 실행 워크플로우

아래 단계를 순서대로 따른다.

### Step 1: 주제와 의도를 고정한다

먼저 아래를 파악한다.

1. 이 글이 설명하려는 핵심 주제가 무엇인지
2. 독자가 왜 이 글을 읽어야 하는지
3. 글이 개념 정리인지, 안티패턴 정리인지, 구현 기법 소개인지
4. 사용자가 직접 준 코드/메모/초안이 있는지

요청이 모호하면 글을 쓰기 전에 최소한 다음 중 하나를 정한다.

- 핵심 개념 1개
- 독자가 얻어갈 결론 1개
- 예시로 쓸 코드 또는 프롬프트 2~3개

### Step 2: 저장소 스타일을 맞춘다

`references/blog-style.md`를 읽고 아래 스타일을 고정한다.

- 정의로 시작하는 구조
- 번호 매긴 핵심 항목
- 짧고 직접적인 설명
- 예시 중심 전개
- 결론보다 참고 자료 중심 마무리

### Step 3: 글 구조를 설계한다

기본 구조는 아래 중 하나를 고른다.

#### A. 개념 정리형

1. `## [주제]이란?`
2. `### 왜 중요한가?`
3. `### 장점과 단점`
4. `### 핵심 개념 1`
5. `### 핵심 개념 2`
6. `### 참고`

#### B. 규칙 / 안티패턴형

1. `#### 개요`
2. `#### 1. 하지 말아야 할 것`
3. `#### 2. 더 나은 방법`
4. `#### 3. 주의할 점`
5. `#### 4. 관련 API 또는 패턴`
6. `##### 참고`

#### C. 구현 기법 정리형

1. `## [주제]이란?`
2. `### 왜 배워야 할까?`
3. `## 적용 기준`
4. `## 핵심 구현 기법`
5. `### 1. 기법 A`
6. `### 2. 기법 B`
7. `### 3. 기법 C`
8. `**참고 사이트**`

### Step 4: frontmatter를 작성한다

기본 형식은 아래를 사용한다.

```yaml
---
layout: post
title: "..."
author: chanhee.kim
date: YYYY-MM-DD HH:MM
updated_at: YYYY-MM-DD
tags: [tag1, tag2]
image: /files/covers/cover-dev-01.jpg
description: "..."
keywords: "..."
---
```

규칙:

- 파일명은 `YYYY-MM-DD-topic-slug.md`
- `title`은 `[Java]`, `[디자인 패턴]`처럼 접두어를 붙일 수 있다
- `description`은 무엇을 정리하는 글인지 한 문장으로 설명한다.
- `title`과 `description`은 무조건 SEO를 적용시켜 작성한다.
- `keywords`는 제목 확장형 검색어로 작성한다
- `updated_at`은 새 글이면 생략 가능하지만, 리라이트라면 넣는 편이 자연스럽다
- 이미지가 확정되지 않았으면 기존 저장소 패턴 안에서 보수적으로 잡거나 사용자 확인을 남긴다

### Step 5: 본문을 작성한다

본문 작성 시 아래를 적용한다.

- 첫 문단 또는 첫 헤딩에서 개념을 바로 정의한다
- `왜 필요한가`, `언제 쓰는가`, `장점`, `단점`을 먼저 설명한다
- 핵심 내용은 번호 매긴 항목으로 나눈다
- 각 항목에는 가능한 한 예시를 붙인다
- 예시가 필요하면 `나쁜 예`, `좋은 예`, `적용 전`, `적용 후` 형식을 우선한다
- 설명 직후 코드 블록을 배치한다
- 마지막은 `참고`, `참고 사이트`, `읽을거리`로 마무리한다

### Step 6: AI 프롬프트 주제로 변환한다

AI 프롬프트 주제라도 쓰려면 아래 방식으로 변환한다.

- "어떤 모델이 더 좋다"보다 "좋은 프롬프트를 어떻게 구성하는가"를 설명한다
- 비교형 리뷰보다 규칙형, 패턴형, 안티패턴형 글로 푼다
- 프롬프트 예시는 `나쁜 예 / 좋은 예`가 특히 잘 맞는다
- 필요하면 간단한 코드 예시와 함께 붙인다

예시 구조:

1. `## 프롬프트 엔지니어링이란?`
2. `### 왜 중요한가?`
3. `#### 1. 역할을 먼저 지정할 것`
4. `#### 2. 입력 조건을 구체적으로 줄 것`
5. `#### 3. 출력 형식을 명시할 것`
6. `#### 4. 나쁜 프롬프트와 좋은 프롬프트 비교`
7. `##### 참고`

## 출력 형식

요청에 따라 아래 형식 중 하나로 출력한다.

### 1. 아이디어 단계

```markdown
## 글 방향 제안

### 추천 제목
1. ...
2. ...
3. ...

### 추천 구조
1. ...
2. ...
3. ...

### 핵심 메시지
- ...
```

### 2. 개요 단계

```markdown
## 글 개요

### 대상 독자
- ...

### 핵심 질문
- ...

### 섹션 구성
1. ...
2. ...
3. ...
```

### 3. 전체 초안 단계

```markdown
---
layout: post
title: "..."
author: chanhee.kim
date: ...
tags: [...]
image: ...
description: "..."
keywords: "..."
---

## ...

본문

##### 참고
- ...
```

### 4. 수정 단계

수정 요청이면 아래 순서로 처리한다.

1. 기존 글의 핵심 주제 유지
2. 번호 매긴 항목 추가
3. 예시 코드 또는 프롬프트 보강
4. 메타데이터 보완

## 품질 체크리스트

최종 출력 전 아래를 점검한다.

- [ ] 글이 정의나 개요로 바로 시작하는가
- [ ] `왜 중요한가` 또는 동등한 설명이 있는가
- [ ] 번호 매긴 핵심 항목이 있는가
- [ ] 각 항목에 예시, 코드, 또는 프롬프트가 붙어 있는가
- [ ] 최근 스타일의 비교표/요약 대시보드가 기본 구조로 섞이지 않았는가
- [ ] 개인 경험을 지어내지 않았는가
- [ ] 최신성이 필요한 주제는 확인했는가
- [ ] 마지막에 참고 자료가 있는가
